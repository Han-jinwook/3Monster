# 🔱 3Monster 공통 SDK 및 개발 정책 통합 강령 (Integration Charter)

- **문서 번호**: 3M-CHARTER-001
- **최종 버전**: v1.2
- **갱신 일시**: 2026-06-09
- **통제권자**: Monster 총괄 AI (Hub AI)

---

본 강령은 3Monster 패밀리 개별 앱(NPlace-DB, CafeCrawler 등)을 설계하고 통합할 때 반드시 준수해야 하는 **최고 권위의 통합 개발 헌장**입니다. 타 세션의 AI 및 개발자는 본 강령에 기록된 기술 구조와 정책을 최우선으로 준수해야 하며, 상세한 구현부와 사용 현황은 연동 가이드 및 사용대장을 상호 참조하십시오.

---

## 🧭 제1장: 4대 핵심 공통 SDK (Software Development Kit)

3Monster 루트의 `라이브러리/` 폴더에 패키지 형태로 사전 구현되어 있으며, 개별 앱 개발 시 **복사 붙여넣기(Import)**하여 그대로 사용해야 하는 실체적인 파이썬 라이브러리 엔진입니다.

### 1. 🔒 인증 엔진 (`auth.py`) - 라이선스 및 HWID 기기 락킹
*   **핵심 기술**: Supabase REST API 직접 호출, Windows 기기 고유 하드웨어 식별자(HWID) 해싱 및 바인딩.
*   **원칙**:
    - **1PC 1Key 바인딩**: 최초 검증 시 사용자의 기기 HWID를 DB에 영구 귀속시켜 라이선스의 무단 배포 및 부정 사용을 원천 차단합니다.
    - **0.5초 타임아웃**: 서버 장애나 오프라인 상황에서 GUI 멈춤 현상(Deadlock)이 없도록 Supabase 커넥션에 강제로 0.5초 시간제한을 둡니다.
    - **라이선스 이원화 정책 (Deluxe vs Free Trial)**: 
      - 정식 유료 라이선스 및 **디럭스(`DLX-`)** 제품은 시리얼 키 방식으로 `licenses` 테이블을 통해 인증하며, 서버가 반환하는 `collection_limit`(예: 디럭스 1000건)을 작동 한도로 사용합니다.
      - 시리얼 번호가 없는 **순수 무료 체험판**은 `trial_logs`를 조회/업데이트하며, 기기당 **생애 1회 50건**으로 한도를 영구 고정합니다. 모든 개별 앱은 이 인증 플로우를 분리하여 구현해야 합니다.

### 2. 🔄 업데이트 엔진 (`updater.py`) - 실시간 버전 대조 및 원격 배포
*   **핵심 기술**: HTTP 백그라운드 스트리밍 다운로드, 원격 패치 배치(Batch) 파일 프로세스 연동.
*   **원칙**:
    - 앱 부팅 시 Supabase의 최신 버전 테이블(`app_versions`)을 0.2초 이내 초고속 검증합니다.
    - 업데이트 감지 시 대화상자를 띄우고, 승인 시 백그라운드로 스트리밍 패치 파일을 내려받은 후 앱 프로세스를 스스로 종료하고 재부팅(자가 덮어쓰기)합니다.

### 3. 📝 로깅 엔진 (`logger.py`) - 5MB 롤링 로그
*   **핵심 기술**: Python `logging.handlers.RotatingFileHandler`.
*   **원칙**:
    - 프로그램 작동 로그 및 Traceback 에러 정보를 기록하되, 사용자 하드디스크의 무분별한 용량 과점유를 막기 위해 **단일 파일 크기 상한선 5MB** 및 백업본 최대 3개 초과 금지를 하드웨어 레벨에서 강제합니다.

### 4. 📁 내보내기 엔진 (`exporter.py`) - 파일 저장 및 탐색기 하이라이트
*   **핵심 기술**: OS 파일 매니저 연동 및 탐색기 선택(Focus) API.
*   **원칙**:
    - **앱별 의미 있는 식별자 전략**: 단순 일시(타임스탬프)를 덧붙이는 구조를 제거하고, 앱별로 가장 식별하기 적절한 고유의 정보(예: NPlace-DB는 `[지역]_[키워드]`, CafeCrawler는 `[카페명]_[수집기간]`)를 호출부에서 넘겨받아 파일명으로 바로 구성합니다.
    - 저장 직후 OS 파일 탐색기를 실행하여 방금 저장된 파일을 파란색으로 선택(Focus)해주어 즉시 사용 가능한 상태를 지원합니다.

## ⚖️ 제2장: 3대 핵심 개발 정책 (Development Policies)

개별 앱이 허브(웹 대시보드)와 상호 유기적으로 작동하고 일관된 브랜드 아이덴티티를 갖도록 규정한 개발 가이드라인입니다. 이는 SDK 코드로 제공되는 것이 아닌 **"구현 규정"**입니다.

### 1. 📂 사용자 설정 및 라이선스 경로 격리 정책 (Two-Track Storage: AppData Isolation)
*   **답변 및 정의**: 본 사안은 별도의 설정을 다루는 공통 SDK를 만들지 않고 **[정책 가이드]**로 통제합니다.
    *   *이유*: 앱마다 다루는 설정(키워드 수집 한도, 카페 URL 리스트, 댓글 템플릿 등)의 구조가 완전히 달라 획일적인 SDK로 구현 시 불필요한 무게만 차지하게 됩니다.
*   **정책**:
    - 모든 개별 앱의 사용자 설정 정보(`user_settings.json`, `templates.json`) 및 라이선스 키 정보 파일(`license.dat`)은 프로그램 설치 경로가 아닌 반드시 윈도우 시스템 공통 경로인 **`%APPDATA%/MarketingMonster/{AppName}/`** 경로에 격리 저장해야 합니다.
    - 이렇게 분리해야만 `updater.py`에 의해 프로그램 설치 폴더 전체가 최신 파일로 덮어쓰기 및 업데이트되거나 사용자가 수동 업데이트를 위해 폴더를 통째로 지우고 새로 다운받아도, 사용자 세팅과 라이선스 인증 정보가 영구 보존(재인증 번거로움 해결)됩니다.

### 2. 🖥️ 화면 해상도 및 배치 오프셋 정책 (Window Geometry Policy)
*   **정책**:
    - **인증 창**: `auth_gui.py`가 구현하는 최초 라이선스 인증 창은 무조건 모니터 화면의 **정중앙(Center)**에 노출되어 사용자의 시선을 집중시킵니다.
    - **메인 대시보드**: 개별 앱 메인 대시보드는 해상도 **`900x700`** 표준 크기를 적용하며, 기동 시 모니터 좌상단 모서리로부터 가로 40px, 세로 40px of the 오프셋 간격(`+40+40`)을 두고 출력해야 합니다. 브라우저 창이나 다른 활성 윈도우와 완전히 겹쳐 숨겨지는 문제를 미연에 방지합니다.

### 3. 📋 에러 발생 시 무의존성 클립보드 복사 정책 (Clipboard Error Copy Policy)
*   **정책**:
    - 프로그램 오작동 및 치명적 에러 발생 시, 외부 파이썬 클립보드 패키지 의존성 없이 표준 Tkinter API를 백그라운드로 제어하여 **에러 전문과 하드웨어 HWID 사양이 결합된 디버그 리포트**를 사용자의 클립보드에 실시간 강제 복사합니다.
    - 팝업 창을 통해 사용자에게 "에러 내용이 복사되었습니다"를 고지한 후 3Monster 통합 허브의 1:1 고객센터 링크로 직접 유도하여 유저가 `Ctrl + V` 한 번만으로 정밀 에러 접수를 할 수 있도록 UX를 일원화합니다.

## 💾 제3장: 무결성 데이터 저장 및 관리 정책 (Zero-maintenance Data Policy)

모든 개별 앱의 로컬 데이터(DB, CSV, 로그) 관리와 UI 버튼 배치는 사용자 혼란(컴맹 기준)을 0%로 만들기 위해 아래의 단일화 정책을 100% 준수해야 합니다.

### 1. 트랙 1: 내 문서 고정 절대 경로 저장 정책 (Data & Logs)
*   모든 수집 결과 데이터 및 텍스트 로그는 무조건 윈도우 표준 경로인 **`내 문서(Documents)\MarketingMonster\[프로그램명]`**에 고정적으로 저장합니다.
*   유저가 다운받은 압축파일(ZIP)을 바탕화면, D드라이브, USB 등 어디에 풀고 실행하든 상관없이 저장 경로는 내 문서로 일원화되어 권한 오류(WinError 5 등)와 경로 꼬임을 원천 차단합니다.
*   사용자 설정 정보(`user_settings.json`) 및 라이선스(`license.dat`)는 본 문서 제2장 제1항에 의거하여 **`%APPDATA%\MarketingMonster\[프로그램명]` (트랙 2)**에 보관하므로 서로 영향을 주지 않습니다.

### 2. 의미 있는 식별자 중심 파일 관리
*   의미 없는 시간 정보(예: `_0617_1320`)의 단순 누적은 파일 개수만 늘리고 직관성을 해치므로 제외합니다.
*   대신 앱마다 고유한 비즈니스 키워드(수집 기간, 지역 및 업종 등)를 식별자로 파일명을 결정합니다. 시간 정보는 운영체제가 제공하는 파일 메타데이터(생성일, 수정한 날짜)를 활용합니다.

### 3. "데이터 초기화" 및 "다운로드" 버튼 금지
*   **초기화/삭제 버튼 금지**: 데이터가 누적되므로 데이터 초기화 버튼을 UI에 배치하지 않습니다.
*   **CSV 다운로드 버튼 금지**: 크롤러가 수집한 데이터는 즉각 백그라운드에서 SQLite DB에 안전하게 적재되므로, 별도의 "CSV 다운로드" 버튼을 UI에 노출하지 않습니다.

### 4. [📁 작업 폴더 열기] 버튼 = 단일 트리거 통일
*   데이터와 관련된 UI 버튼은 오직 **`[📁 작업 폴더 열기]`** 단 하나만 배치합니다.
*   이 버튼 클릭 시 다음과 같은 연쇄 동작이 백그라운드에서 일어납니다:
    1.  가장 최근 누적된 DB 파일을 바탕으로 **즉시 CSV 결과물을 생성(Export)**합니다.
    2.  `내 문서\MarketingMonster\[프로그램명]` 폴더를 윈도우 탐색기로 즉각 띄워줍니다.
    3.  이때 복잡한 원본 DB와 로그 파일은 해당 폴더 하위의 `DB/` 및 `Logs/` 서브 폴더에 조용히 숨겨둡니다. 
*   결과적으로 사용자는 버튼 하나만 누르면 바로 엑셀(CSV) 결과물을 눈앞에서 볼 수 있는 압도적으로 심플한 UX를 경험합니다.

## 📦 제4장: 깔끔한 패키징 및 배포 정책 (Clean Packaging Policy)

배포용 압축 파일(ZIP)을 구성할 때, 일반 사용자가 압축을 풀고 무엇을 실행해야 할지 1초 만에 알 수 있도록 **루트 폴더의 시각적 복잡도를 최소화**해야 합니다.

### 1. 단일 실행파일 노출 원칙
*   사용자가 ZIP 압축을 해제했을 때 최상단(Root) 폴더에는 **메인 실행 파일(`.exe`) 단 하나와 `_internal` 폴더만 보이도록** 구성하는 것을 표준으로 삼습니다.
*   `_internal` 폴더는 PyInstaller 등에 의해 생성되는 종속성 폴더이며, 윈도우 기본 설정상 일반 폴더와 다르게 눈에 잘 띄지 않거나 사용자가 무시하기 쉽습니다.

### 2. 잔여 파일 노출 금지
*   박쥐 파일(`.bat`), 파이썬 스크립트(`.py`), 텍스트 설명서(`.txt`), 설정 파일(`.json`) 등 시스템 파일이나 부수적인 파일들이 **절대로 루트 폴더에 너저분하게 노출되어서는 안 됩니다.**
*   필요한 모든 에셋과 설정은 `_internal` 내부에 은닉하거나 실행 파일 내부에 빌트인(Built-in)하여 패키징해야 합니다.
