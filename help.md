# 고객센터 알림 및 이동 관련 기술 명세서 (help.md)

본 문서는 고객센터 알림 리스트에서 알림을 클릭했을 때 발생하는 페이지 이동 오류, 자동 선택 및 전개(점핑) 오작동, 화면 깜박임(Flickering) 현상의 원인과 이를 안정적으로 해결하기 위한 기술적 해결책을 상세히 정리합니다.

---

## 1. 현재 발생한 주요 문제점 (Symptoms)

1. **알림 클릭 시 자동 선택(점핑) 실패 및 고객센터 페이지 멈춤 현상**:
   * 알림 드롭다운에서 일반 문의(버그, 라이선스 등) 알림을 클릭했을 때 `/support?ticket_id=X`로 정상 이동하지만, 이미 고객센터 페이지에 있는 상태에서는 다른 알림을 클릭해도 화면에 아무런 반응이 없고 특정 티켓이 자동으로 열리지 않습니다.
2. **화면 깜박임(Flickering) 발생**:
   * 알림을 클릭했을 때 페이지가 순간적으로 `/support?ticket_id=X`로 갔다가 다시 `/support`로 리다이렉션되면서 화면이 번쩍이고 데이터 점핑이 이루어지지 않는 현상이 발생합니다.
3. **대시보드 내 기술지원 이동 버튼 오류**:
   * 사용자 대시보드(`UserDashboard.tsx`)의 '기술지원 게시판 이동' 버튼 클릭 시, 브라우저 해시 주소만 `#/support`로 변경될 뿐 실제 페이지 전환이 이루어지지 않아 메인 화면에 정체됩니다.

---

## 2. 기술적 원인 분석 (Technical Root Cause)

### 원인 A: URL 쿼리 파라미터 변화에 대한 컴포넌트의 비반응성
* `CustomerSupport.tsx` 내에서 `ticket_id`를 파라미터로부터 읽어와 해당 티켓을 자동 선택하는 `useEffect`가 존재합니다.
* 하지만 이 훅의 의존성 배열(dependency array)이 `[tickets, purchasedLicenses]`로만 설정되어 있어, 이미 동일한 컴포넌트가 마운트된 상태에서 URL의 쿼리 스트링(`?ticket_id=X`)만 바뀔 경우 변경 사항을 전혀 감지하지 못하고 실행되지 않습니다.

### 원인 B: 양방향 동기화 처리 시의 레이스 컨디션 (화면 깜박임 및 리다이렉션 유발)
URL의 쿼리 스트링 상태와 React State(`selectedTicketForDetail`)를 동기화하기 위해 양방향으로 훅을 설정했을 때 치명적인 레이스 컨디션이 발생합니다.
1. 사용자가 알림을 클릭하여 `/support?ticket_id=X`로 진입합니다.
2. 컴포넌트가 최초 렌더링될 때, Supabase로부터 티켓 목록을 가져오는 `fetchTickets()`는 **비동기**로 처리되므로 초기 `tickets` 상태는 빈 배열(`[]`)입니다.
3. 이 시점에 **URL → State 동기화 훅**이 실행되지만, `tickets.length > 0`이 거짓이므로 `selectedTicketForDetail` 상태는 여전히 `null`로 유지됩니다.
4. 동시에 **State → URL 동기화 훅**이 실행됩니다. 이 훅은 `selectedTicketForDetail`이 `null`인데 URL에 `ticket_id=X`가 남아 있는 것을 감지하고, 상태와 URL의 싱크를 맞추기 위해 URL에서 `ticket_id` 파라미터를 강제로 삭제(`navigate('/support', { replace: true })`)해 버립니다.
5. 결과적으로 티켓 데이터가 다 로드되기도 전에 URL의 `ticket_id` 파라미터가 유실되며 화면 깜박임(리다이렉션)이 발생하고, 이후 비동기 조회가 완료되어 `tickets` 배열이 채워지더라도 URL에 파라미터가 없기 때문에 자동 선택(점핑)이 수행되지 않습니다.

### 원인 C: `BrowserRouter` 환경에서의 잘못된 해시 주소 조작
* 프로젝트는 HTML5 History API 기반의 `BrowserRouter`를 사용하고 있습니다.
* `UserDashboard.tsx`에서 `window.location.hash = '#/support'`를 통해 주소를 변경하면, 라우터가 경로 변화를 감지하지 못하고 단순 해시 플래그로만 인식하여 페이지 전환을 수행하지 못합니다.

---

## 3. 올바른 해결 방안 (Proposed Architecture)

레이스 컨디션을 유발하는 양방향 `useEffect` 체인을 끊어내고, 단방향 흐름과 명시적인 핸들러를 사용하도록 재설계해야 합니다.

### 해결책 1: URL 파라미터 변화를 감지하기 위해 React Router 훅 사용
* `window.location.search` 대신 `react-router-dom`의 `useLocation()` 훅을 사용하여 `location.search` 및 `location.hash`를 받아옵니다.
* 티켓 자동 선택 `useEffect`의 의존성 배열에 `location.search`를 추가하여 URL 변경 시 즉시 선택 로직이 반응하도록 수정합니다.

### 해결책 2: 무조건적인 State-to-URL 동기화 훅 제거 및 이벤트 기반 처리
* 데이터가 비동기로 로딩되는 중에 URL 파라미터를 강제로 삭제하는 `useEffect` 동기화 로직을 제거합니다.
* 대신, 사용자가 UI 상에서 특정 티켓을 직접 클릭하여 선택하거나 닫을 때만 **명시적인 이벤트 핸들러**를 통해 URL을 변경해 줍니다.
  * **목록에서 티켓 클릭 시**: `navigate(\`/support?ticket_id=\${ticket.id}\`, { replace: true })` 호출
  * **상세 보기 닫기 버튼 클릭 시**: `navigate('/support', { replace: true })` 호출
* 이렇게 처리하면 데이터 로딩 시점의 레이스 컨디션이 원천적으로 차단되며 화면 깜박임과 자동 선택 실패 오류가 깔끔하게 해결됩니다.

### 해결책 3: `UserDashboard.tsx` 내비게이션 교정
* `useNavigate` 훅을 사용해 `navigate('/support')`로 주소를 변경하도록 수정합니다.

## 4. UI/UX 이외의 중요 기능 정리 (기능 명세 누락 검토)

프로젝트 진행 과정에서 구현되었으나 공식 문서나 기능 명세에 명확히 기록되지 않은 중요 백엔드 및 상태 관리 비즈니스 로직은 다음과 같습니다.

1. **문의 등록 시 구매자 자동 권한 부여 및 이메일 바인딩**:
   * 일반 사용자가 기술 지원 문의를 작성할 때 기입한 이메일이나 크몽 닉네임을 기반으로 `licenses` 테이블을 조회합니다.
   * 유효한 라이선스가 확인되면 라이선스의 `contact` 이메일을 현재 문의자 이메일로 자동 동기화(바인딩)합니다.
   * 동시에 `users` 테이블에 해당 사용자를 `role: 'buyer'`(구매자) 상태로 생성 또는 갱신(upsert)하여 로그인 시 정상적으로 구매자 대시보드 권한을 가질 수 있도록 자동 승인 처리합니다.
2. **관리자 답변 수정 시 알림 재동기화 설계**:
   * 사용자가 문의 알림을 읽었는지 여부는 `localStorage`에 저장된 마지막 메시지 ID와 DB 상의 thread 마지막 메시지 ID를 비교하여 판별합니다.
   * 관리자가 기존 답변을 수정할 경우 동일한 메시지 ID가 유지되므로 사용자가 이미 읽은 것으로 간주하여 새로운 알림 배지가 활성화되지 않는 문제가 있었습니다.
   * 이를 해결하기 위해 관리자가 답변을 수정할 때 메시지 ID 뒤에 `-edited-{timestamp}` 접미사를 붙여 새로운 고유 ID로 갱신함으로써 사용자 화면에 알림 배지가 다시 실시간으로 나타나도록 구현되어 있습니다.
